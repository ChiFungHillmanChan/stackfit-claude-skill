## Rules For The Implementing Agent

1. Do not substitute services. If a constraint here appears wrong, stop and raise it rather than choosing an alternative.
2. Do not add infrastructure not listed here. Every component cites a requirement; anything absent was deliberately excluded.
3. Build in checklist order. Later steps assume earlier ones.
4. Fields marked `[assumed]` were inferred, not confirmed. Verify before relying on them.

**Note the inverted ratio.** This system is roughly 16:1 write-heavy. Most web architectures assume the opposite, and defaults tuned for read-heavy traffic misprice this system badly.

| Notification | SNS + Twilio | ~2,000 msgs/mo | 30s push delivery |

### The decision that dominates cost

Device firmware **must** batch 10 readings into one MQTT publish on a 5-minute interval.

AWS IoT Core meters in 5KB increments at $1 per million messages. Unbatched at 30-second intervals, 12,000 units generate 1.037 billion messages per month — **$1,037/mo, or 86% of the entire budget**, before any database exists. A 10-reading batch is roughly 1KB and still bills as a single message, cutting that line to $104/mo.

Do this before onboarding any fleet. Retrofitting firmware across a deployed fleet is far more expensive than shipping it correctly.

## Schema

Postgres 16 with the TimescaleDB extension.

```sql
-- Relational core: small, mutable, normal Postgres tables.
CREATE TABLE tenants (
  id          BIGSERIAL PRIMARY KEY,
  name        TEXT NOT NULL,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE sites (
  id          BIGSERIAL PRIMARY KEY,
  tenant_id   BIGINT NOT NULL REFERENCES tenants(id),
  name        TEXT NOT NULL,
  timezone    TEXT NOT NULL
);

CREATE TABLE units (
  id            BIGSERIAL PRIMARY KEY,
  site_id       BIGINT NOT NULL REFERENCES sites(id),
  serial        TEXT NOT NULL UNIQUE,
  unit_type     TEXT NOT NULL,          -- 'freezer' | 'chiller' | 'display'
  min_temp_c    NUMERIC(4,1) NOT NULL,
  max_temp_c    NUMERIC(4,1) NOT NULL,
  commissioned  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- serves: "all units for a tenant" on dashboard load
CREATE INDEX idx_units_site ON units (site_id);

-- Time-series: the hypertable. Narrow columns on purpose — every added
-- column costs ~35GB/yr uncompressed at this write rate.
CREATE TABLE readings (
  time        TIMESTAMPTZ  NOT NULL,
  unit_id     BIGINT       NOT NULL,
  temp_c      REAL         NOT NULL,
  humidity    REAL,
  door_open   BOOLEAN      NOT NULL DEFAULT false,
  battery_pct SMALLINT
);

SELECT create_hypertable('readings', 'time', chunk_time_interval => INTERVAL '1 day');

-- Compression is not optional. Without it, storage is ~15x and the budget
-- breaks by month 4.
ALTER TABLE readings SET (
  timescaledb.compress,
  timescaledb.compress_segmentby = 'unit_id',
  timescaledb.compress_orderby   = 'time DESC'
);
SELECT add_compression_policy('readings', INTERVAL '7 days');

-- serves: "readings for one unit over a time range" — audit export and
-- unit detail chart. Composite because every query filters unit then time.
CREATE INDEX idx_readings_unit_time ON readings (unit_id, time DESC);

-- Continuous aggregate. Alerting and charts read THIS, never raw readings.
CREATE MATERIALIZED VIEW readings_5min
WITH (timescaledb.continuous) AS
SELECT
  time_bucket('5 minutes', time) AS bucket,
  unit_id,
  avg(temp_c)  AS avg_temp_c,
  max(temp_c)  AS max_temp_c,
  min(temp_c)  AS min_temp_c,
  bool_or(door_open) AS door_opened,
  count(*)     AS sample_count
FROM readings
GROUP BY bucket, unit_id;

SELECT add_continuous_aggregate_policy('readings_5min',
  start_offset => INTERVAL '1 hour',
  end_offset   => INTERVAL '1 minute',
  schedule_interval => INTERVAL '1 minute');

-- Current state. 12,000 rows, upserted by the ingest worker. This is why
-- there is no Redis at this tier.
CREATE TABLE unit_current_state (
  unit_id     BIGINT PRIMARY KEY REFERENCES units(id),
  last_seen   TIMESTAMPTZ NOT NULL,
  temp_c      REAL NOT NULL,
  door_open   BOOLEAN NOT NULL,
  battery_pct SMALLINT,
  in_excursion BOOLEAN NOT NULL DEFAULT false
);

-- serves: "show me every unit currently in excursion" — the primary
-- dashboard query, partial index because the true set is small
CREATE INDEX idx_current_excursion ON unit_current_state (unit_id)
  WHERE in_excursion = true;

CREATE TABLE excursions (
  id          BIGSERIAL PRIMARY KEY,
  unit_id     BIGINT NOT NULL REFERENCES units(id),
  started_at  TIMESTAMPTZ NOT NULL,
  ended_at    TIMESTAMPTZ,
  peak_temp_c REAL NOT NULL,
  acknowledged_by BIGINT,
  notified_at TIMESTAMPTZ
);

-- serves: "open excursions" for the alert evaluator's dedupe check
CREATE INDEX idx_excursions_open ON excursions (unit_id)
  WHERE ended_at IS NULL;

-- serves: audit export, "all excursions for a tenant in a date range"
CREATE INDEX idx_excursions_started ON excursions (started_at DESC);

-- Retention: 2 years, with tiering to object storage after 90 days.
SELECT add_retention_policy('readings', INTERVAL '2 years');
```

### Index justification

| Index | Serves query | Notes |
|---|---|---|
| `idx_units_site` | All units for a tenant, dashboard load | Small table, plain btree |
| `idx_readings_unit_time` | Unit detail chart, audit export | Composite order matters: filter unit, then range time |
| `idx_current_excursion` | Primary dashboard: units currently in excursion | Partial — true set is a few dozen rows out of 12,000 |
| `idx_excursions_open` | Alert dedupe before paging | Partial, prevents repeat pages for one fault |
| `idx_excursions_started` | Audit export by date range | Compliance queries |

No index on `readings(time)` alone. The hypertable already partitions by time, so it would be redundant and cost write throughput.

|---|---|---|---|---|

| GET | `/api/excursions` | Staff session | 300ms | Open excursions for tenant |
| POST | `/api/excursions/:id/ack` | Staff session | 300ms | Acknowledge, stops escalation |
| GET | `/api/audit/export` | Staff session, admin role | 30s | Async job; must run against the replica at Growth |
| GET | `/healthz` | None | 50ms | Includes ingest lag |

## Caching

No cache tier at MVP. `unit_current_state` is 12,000 rows and serves dashboard reads directly from Postgres.

Adding Redis here would buy an invalidation bug surface for no measured gain. It enters at Growth, when per-reading UPDATEs on the current-state table begin contending with the ingest write path.

When Redis does enter:

| Key pattern | Contents | TTL | Invalidated by |
|---|---|---|---|
| `unit:{id}:state` | Latest reading, door, battery | 15 min | Every ingest write, overwrite not delete |
| `tenant:{id}:excursions` | Open excursion list | 30s | Excursion open/close event |

Cold-start behavior: dashboard falls back to `unit_current_state` in Postgres on a cache miss. No stampede guard needed because the fallback query is indexed and cheap.

## Build Checklist

1. Flash device firmware to batch 10 readings per MQTT publish on a 5-minute interval. This is the difference between $104/mo and $1,037/mo in IoT Core messaging.
2. Create the AWS IoT Core thing type and policy. Register each unit by serial with its own certificate. Never share a credential across the fleet.
3. Provision Tiger Cloud Scale plan, 4 vCPU / 16GB.
4. Apply the full schema above. Create the hypertable with a 1-day chunk interval **before** the first row lands — converting later requires a rewrite.
5. Enable compression and the 7-day compression policy. Verify with `SELECT * FROM hypertable_compression_stats('readings')` after 8 days.
6. Create the `readings_5min` continuous aggregate and its refresh policy.
7. Deploy the ingest worker with a 15-minute local disk buffer. Use batched `COPY`, not row-by-row `INSERT`.
8. Deploy the alert evaluator on a 15s tick reading `readings_5min`. Dedupe against open excursions before paging.
9. Wire SNS for email and Twilio for SMS escalation.
10. Set the tiered-storage policy to move chunks older than 90 days to object storage, retaining 2 years.
11. Add one alert on the pipeline itself: ingest lag above 60s. A monitoring system that fails silently is worse than none.

## Scaling Triggers

| Tier | Move up when | Cost |
|---|---|---|
| MVP (12,000 units) | Sustained write TPS passes 800/s, or ingest lag exceeds 60s during a reconnect storm, or the disk buffer fills before the database recovers | $500/mo |
| Growth (50,000 units) | IoT Core messaging alone passes $500/mo, or write TPS passes 3,000/s, or a single writer saturates during reconnect storms | $1,759/mo |
| Scale (200,000 units) | Re-run the interview with measured numbers rather than extending this diagram | $4,235/mo |

Growth adds Kinesis (durable buffer), Redis (current state), and a read replica. Scale replaces IoT Core with a self-hosted EMQX cluster and Kinesis with Kafka, and tiers audit data to S3 with Athena.

## Assumptions To Verify

| Assumption | Basis | If wrong |
|---|---|---|
| 15x compression ratio | Typical for narrow numeric time-series segmented by unit | At 5x, storage is ~165GB/yr and Tiger Cloud costs roughly $30/mo more. Not budget-threatening, but verify after 30 days of real data. |
| 30-second reading interval | Stated in the interview | At 10s, every cost in this document triples and the MVP tier exceeds budget. This is the single most important number here. |
| 3x reconnect-storm peak | Inferred, not measured | If a regional cellular outage reconnects the whole fleet at once, peak could be 20x+ and the 15-minute disk buffer is undersized. Measure during the first real outage. |
| 2-year retention | HACCP audit window, not confirmed with counsel | A 5-year requirement roughly doubles tiered storage and makes the S3 Parquet archive necessary at MVP rather than Scale. |
| No PII beyond staff email | Inferred from the described use case | If units are in named customer premises with contact details, GDPR-style deletion obligations affect the audit archive design. |
