> System class: **S — Small.** This spec is short because the system is small. That is the correct outcome, not an omission.

## Rules For The Implementing Agent

1. Do not substitute services. If a constraint here appears wrong, stop and raise it.
2. Do not add infrastructure not listed here. There is deliberately no queue, no cache, no CDN beyond what Vercel provides, and no read replica. Each was considered and rejected against the profile.
3. Build in checklist order.
4. Fields marked `[assumed]` were inferred, not confirmed.

## Access Patterns

| Pattern | Query | Rate / shape |
|---|---|---|
| Primary read | Class schedule for a date range, joined to instructor and room | 8/s — relational, the join is the query |
| Primary write | Create booking and decrement remaining capacity | 2/s — must be one transaction |
| Secondary read | One member's upcoming bookings | Point lookup by member id |
| Admin | Instructors manage classes, rosters, cancellations | CRUD, low volume |

**Datastore chosen: Postgres (Supabase).** The primary write must atomically decrement capacity and insert a booking. That transaction, plus a genuinely relational read pattern, is exactly Postgres. A key-value store would mean building the transaction machinery yourself.

## Schema

```sql
CREATE TABLE instructors (
  id     BIGSERIAL PRIMARY KEY,
  name   TEXT NOT NULL,
  email  TEXT NOT NULL UNIQUE
);

CREATE TABLE rooms (
  id       BIGSERIAL PRIMARY KEY,
  name     TEXT NOT NULL,
  capacity SMALLINT NOT NULL CHECK (capacity > 0)
);

CREATE TABLE members (
  id         BIGSERIAL PRIMARY KEY,
  email      TEXT NOT NULL UNIQUE,
  name       TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE classes (
  id             BIGSERIAL PRIMARY KEY,
  instructor_id  BIGINT NOT NULL REFERENCES instructors(id),
  room_id        BIGINT NOT NULL REFERENCES rooms(id),
  starts_at      TIMESTAMPTZ NOT NULL,
  duration_min   SMALLINT NOT NULL,
  total_spots    SMALLINT NOT NULL CHECK (total_spots > 0),
  -- The database refuses to oversell even if application logic has a bug.
  -- This constraint is the actual enforcement of the strong-consistency
  -- requirement; the transaction below is how we cooperate with it.
  remaining_spots SMALLINT NOT NULL CHECK (remaining_spots >= 0),
  cancelled      BOOLEAN NOT NULL DEFAULT false
);

-- serves: the schedule query — "classes in this date range", the single
-- highest-volume read in the system
CREATE INDEX idx_classes_starts ON classes (starts_at)
  WHERE cancelled = false;

CREATE TABLE bookings (
  id          BIGSERIAL PRIMARY KEY,
  class_id    BIGINT NOT NULL REFERENCES classes(id),
  member_id   BIGINT NOT NULL REFERENCES members(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  cancelled_at TIMESTAMPTZ,
  stripe_payment_intent TEXT,
  -- one member cannot book the same class twice
  UNIQUE (class_id, member_id)
);

-- serves: "my upcoming bookings" on the member dashboard
CREATE INDEX idx_bookings_member ON bookings (member_id, created_at DESC)
  WHERE cancelled_at IS NULL;

-- serves: class roster for instructors
CREATE INDEX idx_bookings_class ON bookings (class_id)
  WHERE cancelled_at IS NULL;
```

### The booking transaction

This is the one place strong consistency is not optional. Both statements run in a single transaction:

```sql
BEGIN;
  UPDATE classes
     SET remaining_spots = remaining_spots - 1
   WHERE id = $1 AND remaining_spots > 0 AND cancelled = false;
  -- if 0 rows updated, the class is full: ROLLBACK and return 409

  INSERT INTO bookings (class_id, member_id, stripe_payment_intent)
  VALUES ($1, $2, $3);
COMMIT;
```

Do not read remaining spots, decide in application code, then write. That is a read-modify-write race, and under the Monday release burst it will oversell.

### Index justification

| Index | Serves query | Notes |
|---|---|---|
| `idx_classes_starts` | Schedule for a date range | Partial: cancelled classes are never listed |
| `idx_bookings_member` | Member's upcoming bookings | Partial: cancelled bookings excluded |
| `idx_bookings_class` | Instructor's class roster | Partial, same reason |

## API Surface

REST. GraphQL would add a schema layer and a client runtime for a solo developer serving four query shapes.

| Method | Path | Auth | p99 | Notes |
|---|---|---|---|---|
| GET | `/api/classes?from&to` | Public | 400ms | Cacheable, SSR-rendered for SEO |
| POST | `/api/bookings` | Member session | 400ms | Idempotency key required — members double-tap |
| DELETE | `/api/bookings/:id` | Member session | 400ms | Returns the spot to the class |
| GET | `/api/me/bookings` | Member session | 400ms | Cursor pagination |
| POST | `/api/admin/classes` | Staff session | 400ms | Django-equivalent admin CRUD |
| POST | `/api/webhooks/stripe` | Stripe signature | 1s | Must be idempotent; Stripe retries |

**Rate limiting:** 10 booking attempts per member per minute. The Monday burst is legitimate traffic, so limit per member rather than globally.

**Pagination:** cursor-based on `created_at`. Offset pagination shifts under concurrent inserts.

**Idempotency:** required on `POST /api/bookings` and the Stripe webhook. Both are retried by clients you do not control.

## Caching

None. `unit`-level data does not exist at volume here, the schedule query is indexed and small, and Vercel already edge-caches the SSR-rendered public pages.

Revisit only when a measurement shows the schedule query hurting — not before.

## Build Checklist

1. Create the Supabase project and apply the schema. The `CHECK (remaining_spots >= 0)` constraint is load-bearing — do not omit it.
2. Implement the booking transaction exactly as written above. Do not read-modify-write.
3. Set up Vercel on the **Pro** plan. Hobby forbids commercial use and a studio taking payments is commercial.
4. Enable Supabase connection pooling on day one. Serverless concurrency against a connection-limited Postgres is the most common way this stack falls over.
5. Integrate Stripe Checkout in hosted mode. Card data must never reach your servers.
6. Wire Resend for booking confirmations. The free tier is 3,000/month but also 100/day — the Monday release burst is the day to check against that cap.
7. Add one alert: Supabase database size above 400MB. It is the only number worth watching.
